# Manifesto do Projeto

## Por que existimos

Este projeto nasce para reduzir fricção entre ideia e entrega.
Não buscamos automação pela automação. Buscamos confiança para evoluir sistemas sem medo.

Software não falha por falta de ferramentas — falha por falta de clareza.
Este projeto prioriza clareza operacional antes de sofisticação técnica.

## O que acreditamos

### 1. Simplicidade vence genialidade

Se uma solução exige explicação longa, ela está errada para este contexto.
O sistema deve ser compreensível por um desenvolvedor cansado numa sexta‑feira à noite.

### 2. Convenção acima de configuração

Cada escolha configurável multiplica decisões futuras.
Preferimos poucos caminhos previsíveis do que infinitas possibilidades corretas.

### 3. Evolução segura > perfeição inicial

Arquitetura não é algo projetado — é algo descoberto.
Primeiro tornamos possível mudar. Depois tornamos elegante.

### 4. Automação deve remover ansiedade, não criar dependência

Se quebrar, o desenvolvedor deve entender rapidamente o motivo.
A ferramenta não pode virar uma caixa preta.

### 5. Documentação é parte do código

Código explica o "como".
Documentação explica o "porquê".
Sem o porquê, toda manutenção vira reescrita.

### 6. Padronização é uma feature

Tempo gasto discutindo padrões é desperdício recorrente.
O projeto decide uma vez para não decidir sempre.

## O que NÃO somos

* Não somos um framework genérico
* Não somos um orquestrador universal
* Não somos uma coleção de scripts desconectados

Somos uma base operacional opinativa para projetos reais.

## Como tomamos decisões

1. Remove esforço cognitivo repetido?
2. Reduz chance de erro humano?
3. Facilita onboarding de alguém novo?
4. Torna rollback previsível?

Se a resposta não for majoritariamente "sim", a mudança não entra.

## O objetivo final

Que qualquer desenvolvedor consiga:

* Entender o fluxo em minutos
* Confiar no resultado
* Evoluir sem medo

Quando isso acontecer, o projeto cumpriu seu papel.
