# ADR-004 – Estratégia de Versionamento, Promoção e Hotfix

**Status:** Accepted
**Data:** 2026-02-14
**Decisores:** Maintainers do projeto

---

## Contexto

O projeto **reusable-actions** fornece workflows reutilizáveis de CI/CD para múltiplos repositórios e stacks. Desde o início, alguns problemas estruturais ficaram evidentes:

* Nem todo commit deveria gerar uma nova versão
* Forçar versionamento automático (ex: patch) cria releases artificiais
* PRs automáticos sem release real poluem a `main`
* Hotfixes precisam ser rápidos e seguros, sem quebrar o fluxo padrão
* O pipeline não pode assumir nomes fixos de branches (`main`, `develop`)

Além disso, erros reais ocorreram quando:

* semantic-release não detectou versão
* o pipeline tentou promover código mesmo assim
* branches inválidas como `release-` foram criadas

Era necessário **formalizar decisões arquiteturais** para tornar o projeto previsível, reutilizável e seguro.

---

## Decisão

### 1. Semantic-release como autoridade única de versionamento

* semantic-release é a **única fonte de verdade** para decidir se existe uma nova versão
* Nem todo commit gera release — isso é comportamento esperado
* **Não forçamos patch automaticamente**

📌 Consequência direta:

* Se semantic-release não detectar versão → **não há promoção**

---

### 2. Promoção condicionada à existência de release

**📌 Decisão**

As estratégias de promoção de código **não são genéricas** e **não inferem comportamento implicitamente**.
Cada estratégia declara explicitamente quando a promoção acontece e **se depende ou não de uma nova versão detectada pelo** `semantic-release`.

As estratégias finais são:

| Estratégia       | Comportamento                                                                |
| ---------------- | ---------------------------------------------------------------------------- |
| `to-main`        | Promove para `main` **apenas se** houver nova versão detectada               |
| `to-release`     | Fluxo `develop → release-x.y.z → main`, **somente se** houver nova versão    |
| `always-to-main` | Promove para `main` **sempre**, mesmo quando nenhuma nova versão é detectada |

**🧠 Racional**

- Nem todo merge representa uma entrega versionável
- O semantic-release é a autoridade única sobre versionamento
- Criar PRs sem impacto de release gera:
  - ruído operacional
  - histórico poluído
  - falsa percepção de entrega

Por isso:

- Estratégias `to-main` e `to-release` dependem explicitamente da existência de uma versão
- Caso nenhuma versão seja detectada:
  - o workflow **termina com sucesso**
  - **nenhum PR** é criado
  - o pipeline permanece saudável

A estratégia `always-to-main` existe como exceção consciente, para cenários onde:

- o repositório não versiona pacotes
- o versionamento é externo
- ou a promoção contínua é desejada independentemente de release

---

### 3. Hotfix como exceção controlada

Branches `hotfix/*` ignoram completamente as estratégias acima.

Hotfix:
- ❌ não depende de semantic-release para promover
- ❌ não passa por `develop`
- ✔ promove diretamente para `main`
- ✔ executa CI e quality gates
- ✔ semantic-release decide após o merge se haverá versão

Hotfix **não é uma estratégia de promoção**, é um **override controlado de regras**.

**Isso garante:**

* Velocidade em incidentes
* Segurança técnica
* Coerência com o modelo de versionamento

---

### 4. Branches configuráveis (sem hardcode)

O pipeline **não assume** nomes fixos de branches.

Configuração padrão:

```yaml
develop_branch: develop
main_branch: main
prefix_release_branch: release-
```

Isso torna o projeto:

* Reutilizável
* Compatível com diferentes estratégias organizacionais
* Seguro para múltiplos consumidores

---

### 5. Regras claras por tipo de branch

| Branch      | Comportamento                          |
| ----------- | -------------------------------------- |
| `feature/*` | CI completo + possível promoção        |
| `hotfix/*`  | Override: testes + PR direto para main |
| outros      | Apenas testes                          |

---

## Consequências

### Positivas

* Fluxo previsível e explícito
* Eliminação de PRs e releases inúteis
* Hotfix rápido sem comprometer arquitetura
* Menos acoplamento a convenções específicas
* Maior confiança para consumidores

### Negativas / Trade-offs

* Promoções podem não acontecer mesmo com commits recentes
* Consumidores precisam entender semantic-release
* Estratégia exige documentação clara (README)

Esses trade-offs são considerados aceitáveis em troca de **qualidade, previsibilidade e governança**.

---

## Alternativas consideradas (e rejeitadas)

### Forçar patch em todo merge

❌ Rejeitado:

* Cria versões artificiais
* Quebra versionamento semântico
* Aumenta ruído em changelogs

### Hotfix passando por develop

❌ Rejeitado:

* Lento para incidentes
* Aumenta risco operacional

### PR automático sempre

❌ Rejeitado:

* Polui histórico
* Gera branches inválidas
* Diminui confiança no pipeline
